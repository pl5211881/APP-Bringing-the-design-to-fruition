# 组件化搭建平台需求文档

## 1. 文档信息

- 文档版本：V1.0
- 产品阶段：MVP 需求定义
- 目标用户：前端开发、产品经理、设计/交互同学
- 建议技术方向：React + TypeScript + Ant Design + iframe preview
- 参考对象：云梯平台类组件化搭建与预览工作台

## 2. 背景与问题

当前业务组件通常分散在不同项目、文档、代码仓库或口头约定中，产品和前端在复用组件时存在以下问题：

- 组件能力不透明：不知道已有组件支持哪些字段、状态和业务场景。
- 预览成本高：需要前端临时写页面或接入真实业务链路才能看到效果。
- 配置不统一：同类卡片、消息组件、业务模块的数据结构不一致。
- 联调效率低：产品确认、前端开发、设计验收之间缺少统一预览环境。
- 复用沉淀弱：组件使用说明、mock 数据、配置样例没有统一管理。

因此需要建设一个组件化搭建平台，用统一组件协议承载组件物料、配置表单、实时预览和 JSON 导出能力，让产品和前端可以在一个平台中完成“查找组件、调整配置、预览效果、导出配置”的闭环。

## 3. 产品定位

组件化搭建平台是一个面向前端和产品协作的组件物料与可视化配置工作台。

MVP 阶段不做完整低代码页面搭建器，不做复杂拖拽布局和发布审批，而是优先完成组件级搭建闭环：

1. 组件中心：集中展示可复用组件。
2. 可视化搭建：通过表单配置组件字段。
3. 实时预览：通过 iframe 渲染真实组件效果。
4. JSON 导出：导出业务系统可消费的配置数据。

## 4. 产品目标

### 4.1 业务目标

- 降低组件复用门槛，让产品可以理解组件能力和配置项。
- 降低前端联调成本，让组件数据结构、样式状态和预览环境统一。
- 提升组件沉淀效率，让常用业务组件形成可查、可配、可复用的物料资产。
- 为后续 AI 生成配置、模板管理、平台矩阵和发布审批预留基础。

### 4.2 MVP 成功指标

- 至少接入 10 个核心组件物料。
- 用户可以在 3 分钟内完成一次组件选择、配置、预览和 JSON 导出。
- 配置修改后预览区域可以实时更新。
- 导出的 JSON 可以重新导入并恢复相同预览结果。
- 前端可以基于统一协议新增一个组件，而不需要改动搭建器核心逻辑。

## 5. 用户角色

### 5.1 前端开发

核心诉求：

- 维护组件物料和渲染器。
- 快速验证组件在不同配置和 mock 数据下的展示效果。
- 将平台导出的 JSON 接入业务页面。

典型行为：

- 新增组件注册信息。
- 编写组件 schema 和 mock 数据。
- 在预览页验证边界状态。
- 复制导出配置用于业务开发。

### 5.2 产品经理

核心诉求：

- 查看已有组件能力。
- 调整文案、按钮、图片、业务字段等配置。
- 快速确认组件在真实业务场景下的展示效果。

典型行为：

- 搜索组件。
- 根据业务场景选择组件。
- 修改配置项并截图确认。
- 将导出 JSON 或配置截图交给研发。

### 5.3 设计/交互同学

核心诉求：

- 检查组件视觉表现。
- 确认组件在不同状态、主题和内容长度下的布局表现。
- 辅助推动组件规范统一。

典型行为：

- 查看组件详情。
- 切换浅色/深色主题。
- 测试长文案、空状态、异常状态。
- 给出组件样式优化建议。

## 6. MVP 功能范围

### 6.1 首页工作台

首页作为平台入口，展示平台定位、核心入口和最近使用组件。

功能需求：

- 展示平台名称、说明和核心能力入口。
- 提供入口：组件中心、可视化搭建、预览调试、JSON 导出。
- 展示最近使用组件列表。
- 支持浅色/深色主题切换。
- 主题选择保存到 localStorage。

验收标准：

- 用户进入首页后可以明确看到平台用途。
- 用户可以从首页进入组件中心和可视化搭建页面。
- 主题切换后刷新页面仍能保留主题状态。

### 6.2 组件中心

组件中心用于集中展示和检索组件物料。

组件分类：

- 基础组件：文本、按钮、图片、视频、链接。
- 业务卡片：医生卡片、医院卡片、药品卡片、文件卡片、服务授权横幅。
- AI 消息组件：Markdown、快捷回复、思维链、溯源卡、推荐卡。
- 流式组件：SSE 消息、Markdown 消息、CodeBlock 卡片。

组件列表字段：

- 组件名称
- 组件 key
- 组件分类
- 组件描述
- 当前版本
- 适用场景
- 操作入口：查看详情、进入搭建、复制示例 JSON

组件详情字段：

- 组件基础信息
- 配置项说明
- 默认 mock 数据
- JSON 示例
- 预览效果
- 接入说明

功能需求：

- 支持关键词搜索。
- 支持分类筛选。
- 支持点击组件进入详情页。
- 支持从详情页进入可视化搭建。
- 支持复制默认 JSON 示例。

验收标准：

- 至少展示 10 个可用组件。
- 搜索结果能根据组件名称、key、描述匹配。
- 每个组件都必须有 schema、mockData 和预览入口。

### 6.3 可视化搭建

可视化搭建页面是 MVP 核心页面，负责组件选择、字段配置和实时预览。

页面结构：

- 左侧：组件物料列表。
- 中间：iframe 预览区域。
- 右侧：属性配置面板。
- 底部或右下：JSON 查看和导出区域。

功能需求：

- 用户点击左侧组件后，中间预览加载该组件默认效果。
- 右侧根据组件 schema 自动生成表单。
- 用户修改配置表单后，预览区域实时更新。
- 支持恢复默认配置。
- 支持查看当前 JSON。
- 支持复制 JSON。
- 支持下载 JSON 文件。
- 支持导入 JSON 并恢复配置。

字段类型：

- text：单行文本。
- textarea：多行文本。
- number：数字。
- boolean：开关。
- select：下拉选择。
- image：图片地址。
- json：复杂 JSON 编辑。

验收标准：

- 修改配置后 300ms 内触发预览更新。
- 必填字段为空时给出明确提示。
- JSON 导入失败时显示错误原因。
- 导出的 JSON 再次导入后，预览效果保持一致。

### 6.4 iframe 预览运行时

iframe 预览运行时用于隔离组件渲染环境，并模拟真实业务页面。

父页面向 iframe 发送配置：

```ts
{
  type: "COMPONENT_CONFIG_UPDATE",
  data: {
    componentKey: string;
    config: Record<string, unknown>;
    mockData: Record<string, unknown>;
    theme: "light" | "dark";
  }
}
```

iframe 加载完成后回传：

```ts
{
  type: "PREVIEW_FRAME_READY"
}
```

iframe 高度变化后回传：

```ts
{
  type: "PREVIEW_HEIGHT_CHANGE",
  height: number
}
```

运行逻辑：

1. iframe 初始化后发送 `PREVIEW_FRAME_READY`。
2. 父页面收到 ready 后下发当前组件配置。
3. iframe 根据 `componentKey` 查找对应 renderer。
4. renderer 使用 `config`、`mockData`、`theme` 渲染组件。
5. 内容高度变化后 iframe 回传 `PREVIEW_HEIGHT_CHANGE`。

验收标准：

- iframe 首次加载完成后必须发送 ready 消息。
- 父页面切换组件后，iframe 能正确重新渲染。
- renderer 异常时显示兜底错误状态，不影响父页面。
- iframe 内容高度变化时父页面容器高度同步更新。

### 6.5 JSON 导入导出

导出 JSON 是业务接入的核心产物。

导出结构：

```ts
type ComponentExportConfig = {
  componentKey: string;
  version: string;
  config: Record<string, unknown>;
  mockData: Record<string, unknown>;
  theme: "light" | "dark";
  createdAt: string;
};
```

功能需求：

- 支持复制当前 JSON。
- 支持下载 `.json` 文件。
- 支持从 JSON 文本导入。
- 导入时校验 `componentKey` 是否存在。
- 导入时校验基础结构是否合法。

验收标准：

- 复制到剪贴板后给出成功提示。
- 下载文件名包含组件 key 和日期。
- 导入非法 JSON 时显示明确错误。
- 导入不存在的组件 key 时显示组件不存在。

## 7. 核心数据协议

### 7.1 组件元信息

```ts
type ComponentMeta = {
  key: string;
  label: string;
  category: string;
  description: string;
  icon: string;
  version: string;
  schema: ComponentSchema;
  mockData: Record<string, unknown>;
  renderer: React.ComponentType<ComponentRenderProps>;
};
```

### 7.2 组件配置 schema

```ts
type ComponentSchema = {
  fields: Array<{
    name: string;
    label: string;
    type: "text" | "textarea" | "number" | "boolean" | "select" | "image" | "json";
    required?: boolean;
    defaultValue?: unknown;
    options?: Array<{ label: string; value: string }>;
  }>;
};
```

### 7.3 组件渲染参数

```ts
type ComponentRenderProps = {
  config: Record<string, unknown>;
  mockData: Record<string, unknown>;
  theme: "light" | "dark";
};
```

### 7.4 新增组件接入要求

每个新组件必须提供：

- `ComponentMeta`
- `schema`
- `mockData`
- `renderer`
- 默认预览配置
- 至少一个异常或空状态样例

## 8. 页面结构

### 8.1 路由规划

- `/`：首页工作台。
- `/components`：组件中心。
- `/components/:componentKey`：组件详情。
- `/builder`：可视化搭建。
- `/preview-frame`：iframe 预览运行时。
- `/preview/:componentKey`：单组件预览。
- `/export`：JSON 导入导出。

### 8.2 导航结构

MVP 顶部导航：

- 首页
- 组件中心
- 可视化搭建
- JSON 导出

后续版本导航：

- 流式解析器
- 组件雷达
- AI Playground
- 平台矩阵
- 模板管理

## 9. 非功能需求

### 9.1 性能

- 首屏加载控制在 3 秒内。
- 配置变更到预览更新控制在 300ms 内。
- 组件列表 100 个以内时搜索和筛选无明显卡顿。

### 9.2 可维护性

- 新增组件不需要修改搭建器核心逻辑。
- 组件配置表单必须由 schema 自动生成。
- iframe 通信协议必须集中定义，避免散落在业务代码中。

### 9.3 兼容性

- 优先支持 Chrome 最新稳定版。
- 桌面端优先，暂不要求移动端编辑体验。
- 预览区域需要模拟移动端宽度。

### 9.4 安全性

- JSON 导入只解析数据，不执行脚本。
- iframe 消息需要校验 message type。
- 后续接入服务端时再补充权限、鉴权和审计。

## 10. 边界与暂不支持

MVP 暂不支持：

- 完整页面拖拽排版。
- 多人协作。
- 服务端保存。
- 权限审批。
- 正式发布链路。
- AI 自动生成完整页面。
- 复杂工作流编排。
- 运营端直接发布到线上。

## 11. 验收清单

- 首页可以访问，并能进入组件中心和搭建页面。
- 组件中心至少展示 10 个组件。
- 每个组件都有详情、schema、mockData 和预览。
- 可视化搭建页能选择组件并生成配置表单。
- 修改配置后 iframe 预览实时更新。
- iframe 能发送 ready 和 height change 消息。
- JSON 可以复制、下载、导入。
- 非法 JSON、缺失组件、renderer 异常都有兜底提示。
- 主题切换后主页面和 iframe 预览状态一致。
- 新增组件只需注册元信息和 renderer，不需要改搭建器主流程。

## 12. 测试场景

### 12.1 组件注册表测试

- 所有组件都有唯一 key。
- 所有组件都有 label、category、version。
- 所有组件都有 schema、mockData、renderer。
- 缺少必要字段时开发环境给出错误提示。

### 12.2 表单生成测试

- text 字段能正常输入。
- textarea 字段能正常输入长文本。
- number 字段能限制数字输入。
- boolean 字段能切换开关。
- select 字段能展示 options。
- json 字段能校验合法 JSON。

### 12.3 iframe 通信测试

- iframe 初始化后发送 `PREVIEW_FRAME_READY`。
- 父页面发送 `COMPONENT_CONFIG_UPDATE` 后 iframe 正确渲染。
- iframe 高度变化后发送 `PREVIEW_HEIGHT_CHANGE`。
- 非法 message type 被忽略。

### 12.4 导入导出测试

- 当前配置可以复制为 JSON。
- 下载的 JSON 文件可以重新导入。
- 导入非法 JSON 时提示错误。
- 导入不存在的 componentKey 时提示错误。

### 12.5 主题测试

- 浅色主题下组件展示正常。
- 深色主题下组件展示正常。
- 刷新后主题状态保留。

## 13. 版本路线图

### V0.1 原型版

目标：完成可演示的组件配置和预览闭环。

范围：

- 首页静态结构。
- 组件中心静态结构。
- 可视化搭建三栏布局。
- 接入 5 个示例组件。
- 支持本地 mock 数据预览。
- 支持复制 JSON。

### V0.2 MVP 版

目标：形成可供前端和产品实际使用的最小版本。

范围：

- 接入 10 个核心组件。
- 完成 schema 驱动表单。
- 完成 iframe 预览通信。
- 支持主题同步。
- 支持高度同步。
- 支持 JSON 导入、复制、下载。
- 完成基础测试。

### V0.3 可用版

目标：提升组件查找、复用和协作效率。

范围：

- 增加组件搜索。
- 增加分类筛选。
- 增加最近使用。
- 增加组件详情页。
- 增加组件版本字段。
- 增加导入后的配置校验结果展示。

### V1.0 平台版

目标：从本地配置工具升级为团队级组件搭建平台。

范围：

- 服务端保存配置。
- 模板管理。
- 权限与发布审批。
- AI 生成配置入口。
- 组件使用数据统计。
- 平台矩阵。
- 组件雷达。
- 流式解析器。

## 14. 默认假设

- 第一版优先服务前端和产品，不面向运营直接发布。
- 第一版只做组件/卡片级搭建，不做完整页面拖拽排版。
- 第一版配置保存在浏览器本地，暂不接后端。
- 技术实现默认使用 React、TypeScript、Ant Design 和 iframe preview。
- 组件协议优先保证可读、可导出、可复用，不追求复杂低代码 DSL。
